# 実機到着後 現地検証チェックリスト

Mock映像でのテストはすべて完了していますが、以下は実機(カメラ・業者の
USBボタン・実際のタイマーPC)が無いと検証できていません。届き次第、
この順番で確認してください。

## フェーズ1: カメラ単体

- [x] `RealCameraSource`でカメラが認識されるか(`python3 main.py --camera 0`)
      → 確認済み。Logicool C922 Pro Stream Webcamで認識・起動OK。
- [x] ログに表示される実際の解像度・fpsが720p/60fpsになっているか
      → **実測で問題発見・修正済み**: 当初はログ上「60fps」表示でも実際の
      フレーム供給は約29fpsしか出ていなかった(OpenCVのCAP_DSHOWバックエンド+
      FOURCC設定順序の問題)。`src/camera_source.py`を修正し、
      `CAP_MSMF`バックエンド+FOURCCを解像度/fpsより先に設定する順序に
      変更したところ、実測55〜60fpsで安定するようになった。
      30分間の通し確認でも平均約58fps(一部区間で24fps程度への低下が
      1回あり、フェーズ2の追加確認項目とする)。
- [ ] 同じPCのブラウザで監査画面を開き、コマ送り・スワイプ操作が
      実際の映像で違和感なく動くか(★人間による目視確認が必要)

## フェーズ2: 負荷確認(最重要)

- [x] 720p/60fpsの連続キャプチャ+H.264エンコードを、実際に使うPCで
      30分以上連続稼働させ、フレームドロップが起きないか確認する
      → 30分間の自動テストで実施。`frame_count`は概ね安定して増加
      (平均58.16fps)。ただし**終了間際の約90秒間、フレームレートが
      24fps程度まで一時的に低下する事象を1回観測**。エラー・警告は
      発生せず、録画・Webサーバーとも継続動作。USB電源管理等の影響が
      疑われるが未特定。本番前に、より長時間(できれば試合1日分相当の
      数時間)の連続稼働で再現するか確認することを推奨。
- [x] Web監査画面・中央ダッシュボードへのアクセスを同時に行いながら、
      録画側の処理が重くならないか確認する
      → **重大な問題を発見・修正済み**: 長時間の負荷試験中、Web監査
      サーバー(Flask開発用サーバー)が完全に無応答(ハング)になる現象を
      確認した。原因はWerkzeugの1リクエストごとのアクセスログが
      Windows上のコンソール出力(colorama)の内部ロックを介して詰まり、
      全リクエスト処理スレッドが道連れでハングするというもの。
      `src/web_server.py`と`central_dashboard/app.py`でアクセスログの
      出力を抑制する修正を行い、その後の30分間テストではハング再発なし。
      **録画自体はハング中も継続していたため、この問題は現場で誰も
      気づけない可能性が高かった**(合わせて、Webサーバーの起動失敗を
      検知する仕組みも`main.py`に追加済み。詳細は
      VERIFICATION_AFTER_CAMERA.mdを参照)。
- [x] タスクマネージャー(Windows)でCPU使用率・メモリ使用量を監視し、
      時間経過で単調増加していないか(メモリリークの兆候がないか)確認する
      → 30分間で152MB→290MB程度まで増加した後、以降は横ばい
      (顕著な単調増加=メモリリークの兆候は見られなかった)。
      より長時間での傾向は要継続観察。

## フェーズ3: 業者のUSBボタン

- [x] タイマーソフトが排他モード(フルスクリーン専有)で動いていないことを
      業者に確認済み → pynputによるキーフック方式が通用する可能性が高い
      (ただし実機での最終確認は依然として必要。ソフトウェアの実装次第では
      「排他モードではない」といっても独自のキー処理をしていて、他アプリの
      グローバルフックにイベントが渡らないケースもゼロではないため)
- [ ] `tools/detect_keyboard_key.py`を実行し、ボタンが本当にキーボード
      入力として認識されるか、キー名は`f2`で合っているか確認する
- [ ] 業者のタイマーソフトを実際に起動した状態で、
      `python3 main.py --camera 0 --input-mode button`を同じPC上で並行実行し、
      F2キーを押した際に「タイマーソフト側の反応」と「こちらのログ出力」の
      両方が同時に起きるか確認する
- [ ] トグル動作(1回目=スタート、2回目=ストップ)が実際のタイマー操作と
      一致するか確認する
- [ ] 緊急ボタン用に別キーを用意し、`detect_keyboard_key.py`でキー名を
      確認の上、`config.py`の`BUTTON_EMERGENCY_KEY_NAME`に設定する
- [ ] 緊急ボタンを押して、強制的にクリップが生成され、タイマー追跡状態が
      「スタンバイ」にリセットされることを確認する

---

業者確認により、タイマーソフトが排他モードで動いていないことが分かった
ため、もっとも設計の根本に関わるリスクは解消されました。

**フェーズ1・2はカメラ到着後に自動検証済み**(2026-07-16)。60fps未達と
Webサーバーのハングという2つの重大な問題を発見し、いずれも修正済み
(詳細は`VERIFICATION_AFTER_CAMERA.md`参照)。次にリスクが高いのは
**フェーズ3(業者のUSBボタン到着後の同期精度実測)**と、フェーズ4の
障害シナリオ(USBケーブル抜き差し)の実地確認です。いずれも人間の
現場作業が必要な項目です。
