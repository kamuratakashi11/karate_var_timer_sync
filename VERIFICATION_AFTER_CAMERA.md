# カメラ接続確認後の検証リスト

`python3 main.py --camera 0` でカメラが認識され、映像が監査画面に映る
ところまで確認できた後、本番投入前に潰しておくべき項目です。
CHECKLIST.mdのフェーズ2以降を、実際のコマンド付きに具体化したものです。

## A. 画質・解像度の実測確認

- [x] ログの `[RealCameraSource] 実際の設定値: ...` が `1280.0x720.0 @ 60.0fps`
      になっているか確認する
      → **重要な問題を発見・修正済み(2026-07-16)**: ログ表示上は当初から
      「60.0fps」だったが、これはOpenCVの`cap.get(CAP_PROP_FPS)`が
      *要求値*を返していただけで、実際にカメラから配信されていた
      フレームは約29.4fpsしかなかった(`src/camera_source.py`内の
      `RealCameraSource`が使っていたCAP_DSHOWバックエンドの制約と見られる)。
      これを放置していた場合、「やめ」から遡るクリップの実時間範囲が
      想定の約2倍(6〜10秒のつもりが実質12〜20秒相当)になり、通常再生時に
      映像が早送りのように見える、コマ送りの時間感覚がズレる、といった
      形でこのシステムの核心である同期精度が損なわれるところだった。
      `CAP_MSMF`バックエンドに変更し、FOURCC設定を解像度/fps指定より
      先に行う順序に修正した結果、実測55〜60fpsで安定するようになった
      (30分間の通し確認では平均58.16fps。詳細はB参照)。
- [x] 生成されたクリップの実際の解像度・fpsをffprobeで確認する
      → 修正後、`width=1280 height=720 r_frame_rate=60/1`を確認済み。
- [ ] 監査画面(iPad)で実際に技の動き(できれば突き・蹴りなど速い動き)を
      撮影し、コマ送りで「拳が当たっているかどうか」が判定できる解像度・
      鮮明さかどうかを、実際に監査担当者に見てもらって確認する
      (★これは画質の主観判断が必要なため人間の確認が必須です)

## B. 長時間安定性(フレームドロップ・メモリリーク)

- [x] 実際に使うPCで、`python3 main.py --camera 0` を**最低30分、できれば
      1試合分(数時間)相当**連続稼働させる
      → 30分間の自動テストを実施済み(2026-07-16)。`frame_count`・
      メモリ・バッファサイズを30秒おきに自動記録した。
- [x] 稼働中、`/api/health`の`frame_count`が時間経過に比例して増え続けて
      いるか(止まっていないか)、`recording_ok`が常に`true`のままか確認する
      → 60/60回すべて`recording_ok=true`。平均58.16fpsで安定して増加。
      ただし**終了間際の約90秒間だけ24fps程度まで一時的に低下する事象を
      1回観測**(エラーは発生せず、その後停止まで継続動作)。原因は
      未特定(USB電源管理の可能性を疑っているが未確認)。数時間規模の
      連続稼働で再発するか、本番前に確認することを推奨する。
- [x] タスクマネージャー相当(psutilで自動計測)で、python/ffmpegプロセスの
      メモリ使用量を記録し、単調に増加し続けていないか確認する
      → 30分間で約152MB→290MB程度まで増加した後、横ばいで推移。
      明確な単調増加(メモリリークの兆候)は見られなかった。
      数時間規模でも横ばいが続くかは要継続確認。
- [x] `data/buffer/` フォルダのファイル数・合計サイズが、時間が経っても
      一定の範囲(BUFFER_SEGMENTS個程度)に収まり続けているか確認する
      → 30分間、ファイル数10〜11個・合計9.4MB〜12.6MBの範囲で安定。
      増え続ける傾向は見られなかった。

**この過程で、Web監査サーバーが長時間稼働中に無応答になる重大なバグを
発見・修正した(原因: Werkzeugのアクセスログ出力がWindows上でハングを
引き起こす。詳細はE、および`CHECKLIST.md`のフェーズ2を参照)。**

## C. 同期精度の実測(F2ボタン到着後)

**★業者のUSBボタンがまだ到着していないため未着手。到着後、このセクションを
先に進めてください(人間の現場作業が必要です)。**

- [ ] 何か目印になる動作(例: 手を叩く、目立つ物を素早く動かす)をしながら
      F2を押し、生成されたクリップの最終フレーム付近を確認する

  ```bash
  ffmpeg -y -sseof -0.5 -i data/clips/bar_clip_XXXXXX.mp4 \
    -vframes 15 /tmp/last_frames_%02d.png
  ```

  出力された画像を並べて見て、「目印の動作」がクリップの終端付近に
  ちゃんと写っているか確認する(途中で切れていないか)

- [ ] `data/audit_log.jsonl` を確認し、`timer_stop_detected` の
      タイムスタンプと、実際にF2を押した体感タイミングにズレがないか確認する

## D. リングバッファの容量が増え続けないことの確認

- [x] 1時間程度録画を続けた後、`data/buffer/` の合計サイズが増え続けて
      いないか(一定範囲で頭打ちになっているか)確認する
      → 30分間の自動テストでは、ファイル数10〜11個・9.4MB〜12.6MBの
      範囲で頭打ちになっており、増え続ける傾向は見られなかった。
      1時間規模での確認はまだ実施していないが、設計上(FIFOでの
      自動削除)問題が起きる可能性は低いと考えられる。

## E. Webサーバー・iPad同時アクセス時の負荷

- [x] 録画中に、監査画面へのアクセスを継続的に行いながら、録画側の
      フレームドロップが増えないか確認する
      → **重大な問題を発見・修正済み(2026-07-16)**: 定期的なアクセス
      (死活監視ポーリング相当)を続けながらの負荷試験中、録画自体は
      継続する一方でWeb監査サーバーが完全に無応答になる現象を確認した。
      原因はWerkzeugの1リクエストごとのアクセスログ出力がWindows上の
      コンソール出力(colorama)の内部ロックを介して詰まることによる
      ハング。`src/web_server.py`・`central_dashboard/app.py`で
      アクセスログの出力を抑制する修正を行い、修正後は10分間・30分間の
      再テストいずれもハング再発なし(クリップ配信含め全リクエスト正常)。
      **この問題は録画自体は止まらないため、実際の大会中に発生していても
      誰も気づけなかった可能性が高い。iPadの操作(または今回の
      中央ダッシュボードのポーリング)が多いほど発生しやすいと考えられる
      ため、優先度の高い修正だった。**
- [ ] 実際にiPadの実機ブラウザで、クリップ動画の再生・スワイプ操作を
      行った際の体感(カクつき等)を確認する(★人間による確認が必要)

## F. 障害シナリオの実地確認

**★物理的にUSBケーブルを抜き差しする必要があるため人間の現場作業が
必須です。コードを読む限り、`recorder.py`の`_capture_loop`は読み取り
失敗を検知して警告は出すが、カメラが復帰しても`VideoCapture`を
再オープンする処理が無いため、README/CHECKLISTの想定通り「自動復帰は
おそらくしない」設計のままです。挿し直しても復帰しない場合は
手動再起動が必要になる前提で運用手順を用意しておくことを推奨します。**

- [ ] 録画中にUSBケーブルを引き抜き、監査画面に警告(⚠マーク)が
      数秒以内に表示されるか確認する
- [ ] USBケーブルを挿し直した際、録画が自動的に復帰するか、それとも
      手動での再起動が必要か確認する
